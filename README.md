# DOMS — Dynamic Order Management System

نظام إدارة طلبات ديناميكي مبني على الويب يسمح بتصميم نماذج مخصصة وإدارة الطلبات بدون الحاجة للبرمجة.

## 📋 نظرة عامة

DOMS هو تطبيق ويب تفاعلي يوفر:
- ✨ **تصميم نماذج ديناميكي** - أضف حقول مخصصة بأنواع مختلفة
- 📊 **إدارة طلبات متكاملة** - عرض، بحث، فلترة، وتصدير
- 💾 **تخزين مرن** - محلي (IndexedDB + localStorage احتياطي) أو عبر Supabase السحابي
- 📱 **تصميم متجاوب** - يعمل على جميع الأجهزة
- 🌐 **دعم كامل للغة العربية**

يعمل النظام **Offline-First**:
- عند انقطاع الإنترنت، يتم حفظ الطلبات محلياً.
- عند عودة الإنترنت، يتم دفع التغييرات (insert/update/delete) للسحابة ثم سحب ودمج التغييرات من السحابة.

## 🚀 المميزات الرئيسية

### 📝 تصميم النماذج
- **حقول ديناميكية**: نص، رقم، بريد إلكتروني، هاتف، تاريخ، اختيار، متعدد الاختيارات، منطقي، رابط، نص طويل
- **خصائص الحقل**: تحديد الإلزامية، الظهور في النموذج/الجدول/البحث/الفلترة
- **ترتيب مخصص**: إعادة ترتيب الحقول بالسحب والإفلات
- **المفتاح التقني (slug)**: اسم فريد لكل حقل يستخدم في قاعدة البيانات

### 📊 إدارة الطلبات
- **عرض جدولي**: عرض جميع الطلبات في جدول منظم
- **بحث شامل**: البحث في جميع الحقول والبيانات
- **فلترة متقدمة**: فلترة حسب الحالة والحقول المخصصة
- **تصدير البيانات**: تصدير إلى CSV و JSON
- **حالات الطلب**: جديد، قيد المعالجة، مكتمل، ملغي

### 💾 التخزين
- **محلي**:
  - التخزين الأساسي: **IndexedDB**
  - نسخة احتياطية: **localStorage**
- **سحابي**: التكامل مع Supabase لقواعد البيانات السحابية
- **نسخ احتياطي**: تصدير واستيراد البيانات

## 📦 التثبيت والتشغيل

### التشغيل المحلي
1. **تحميل المشروع**
   ```bash
   git clone <repository-url>
   cd مكتبه
   ```

2. **تشغيل الخادم المحلي**
   ```bash
   # استخدام Python
   python -m http.server 8000
   
   # أو استخدام Node.js
   npx serve .
   
   # أو استخدام Live Server في VS Code
   ```

3. **فتح التطبيق**
   - افتح المتصفح على `http://localhost:8000`

ملاحظة: يفضّل فتح المشروع عبر خادم محلي (وليس فتح الملف مباشرة) لأن Service Worker و IndexedDB يعملان بشكل أفضل بهذه الطريقة.

### الإعداد السحابي مع Supabase

1. **إنشاء مشروع Supabase**
   - اذهب إلى [supabase.com](https://supabase.com)
   - أنشئ مشروعًا جديدًا

2. **إنشاء الجداول**
   - افتح SQL Editor في مشروع Supabase
   - نفّذ محتويات `supabase-schema.sql` بالكامل

3. **التأكد من السياسات (RLS)**
   - الملف `supabase-schema.sql` يقوم بتفعيل RLS ويضيف سياسات تسمح بالعمليات على جدول الطلبات.
   - إذا كنت تريد سياسات “لكل مستخدم فقط”، ستحتاج تعديل policies لتستخدم `auth.uid() = user_id`.

4. **إعداد الاتصال داخل التطبيق**
   - في التطبيق: اذهب إلى الإعدادات
   - أدخل Project URL و anon public key
   - فعّل "استخدام Supabase"

5. **تسجيل الدخول**
   - شاشة الدخول تعتمد على Supabase Auth.
   - أنشئ مستخدمين من Supabase Dashboard أو عبر صفحة Auth.

## 🗄️ مخطط قاعدة البيانات

### جدول الحقول (schema_fields)
```sql
- id: المعرّف الفريد
- slug: المفتاح التقني (مثل customer_name)
- label: التسمية المعروضة
- field_type: نوع الحقل
- required: هل الحقل إلزامي
- show_in_form: الظهور في نموذج الإدخال
- show_in_table: الظهور في جدول العرض
- searchable: إمكانية البحث فيه
- filterable: إمكانية الفلترة
- sort_order: ترتيب العرض
- options: خيارات إضافية (JSON)
```

### جدول الطلبات (orders)
```sql
- id: المعرّف الفريد
- user_id: (اختياري) ربط الطلب بالمستخدم
- reference: رمز المرجع الفريد
- status: حالة الطلب
- created_at: تاريخ الإنشاء
- updated_at: تاريخ آخر تعديل (مهم للمزامنة)
- data: البيانات الديناميكية (JSON)
```

ملاحظة: النظام يعتمد على `updated_at/updatedAt` لقاعدة **آخر تعديل يفوز (Last-Write-Wins)** عند الدمج.

## 🔄 المزامنة (Offline / Online) — كيف تعمل

### مصادر البيانات
- **محلياً**: IndexedDB هو المصدر الأساسي.
- **سحابياً**: Supabase جدول `orders` و `schema_fields`.

### نموذج المزامنة
- عند إضافة/تعديل/حذف طلب:
  - يتم تطبيق التغيير محلياً أولاً.
  - إذا كنت Offline أو فشل الاتصال بالسحابة: يتم إضافة العملية إلى `sync_queue`.
- عند عودة الاتصال أو عند Sync:
  - `sync.js` يدفع عناصر `sync_queue` للسحابة (insert/update/delete).
  - ثم يسحب الطلبات من السحابة ويقوم بدمجها مع المحلي عبر `storage.mergeRemoteOrders`.
  - يتم استدعاء `window.DOMS.onSyncComplete` لإعادة تحميل الطلبات والواجهة.

### الدمج والحذف عبر الأجهزة
- الدمج يضيف/يحدّث حسب الوقت (`updatedAt`).
- الحذف عبر الأجهزة يتم اكتشافه عندما:
  - جهاز B يسحب من السحابة ولا يجد الطلب الذي كان موجوداً محلياً (بعد أن تم حذفه من جهاز A)، فيقوم بحذفه محلياً أثناء `mergeRemoteOrders`.

## ✅ اختبار المزامنة بين جهازين (مهم)

1. افتح التطبيق على **جهاز A** و **جهاز B** بنفس إعدادات Supabase.
2. على جهاز A:
   - أضف طلب جديد.
3. على جهاز B:
   - انتظر المزامنة أو أعد تحميل الصفحة.
   - يجب أن يظهر الطلب.
4. اختبار Offline → Online:
   - على جهاز A: اقطع الإنترنت.
   - أضف طلب.
   - أعد الإنترنت.
   - راقب أن المزامنة تتم وأن الطلب يصل للسحابة ثم يظهر على جهاز B.
5. اختبار الحذف:
   - احذف طلب من جهاز A.
   - جهاز B يجب أن يختفي منه بعد المزامنة.

## 🐛 حل المشاكل

### مشاكل شائعة
1. **لا تظهر البيانات**: تحقق من الاتصال بالإنترنت وإعدادات Supabase
2. **الحفظ لا يعمل**: تأكد من تعبئة جميع الحقول الإلزامية
3. **البحث لا يجد**: تحقق من إعدادات الحقول القابلة للبحث

4. **Service Worker Error: `Response body is already used`**
   - يحدث عادة عند محاولة `response.clone()` بعد استهلاك الـ response.
   - تم إصلاحه في `service-worker.js` عبر عمل clone فوراً قبل عمليات الكاش.

5. **`/favicon.ico` 404**
   - هذا تحذير فقط (غير مؤثر على عمل النظام). إذا أردت، أضف ملف `favicon.ico` بجانب `index.html`.

### تصحيح الأخطاء
- افتح أدوات المطور في المتصفح (F12)
- تحقق من console للأخطاء
- راقب علامة التبويب Network للطلبات
